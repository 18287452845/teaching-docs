# 任务一 理解内网安全与网络分段

## 理论知识

### 什么是内网

**内网（Intranet）** 是组织内部使用的专用网络。它通常使用私有 IP 地址，并通过路由器、防火墙或代理服务与其他网络连接。

RFC 1918 定义了三段私有 IPv4 地址：

| 地址块 | 前缀长度 | 地址范围 |
| --- | --- | --- |
| `10.0.0.0/8` | `/8` | `10.0.0.0` - `10.255.255.255` |
| `172.16.0.0/12` | `/12` | `172.16.0.0` - `172.31.255.255` |
| `192.168.0.0/16` | `/16` | `192.168.0.0` - `192.168.255.255` |

私有地址不会在公共互联网中直接路由，但这并不表示使用私有地址的主机天然安全。只要存在错误配置、恶意软件、弱口令、越权账号或不受控的转发通道，风险仍然可能在内网中扩散。

### 为什么内网不能默认可信

传统网络经常形成一种“边界安全”思维：只要在互联网出口部署防火墙，进入内网的设备就可以互相信任。

这种思路存在明显缺陷：

| 风险来源 | 示例 | 可能后果 |
| --- | --- | --- |
| 外部攻击进入内网 | Web 服务漏洞、弱口令、钓鱼邮件 | 攻击者获得第一台内网主机的访问权限 |
| 内部终端被感染 | 恶意软件、勒索软件、供应链风险 | 恶意程序主动扫描和连接其他服务器 |
| 权限使用不规范 | 域管理员登录普通终端 | 高权限凭据被窃取 |
| 网络过度开放 | 办公终端可直接访问服务器 SMB、RDP | 单点失陷后攻击范围扩大 |
| 审计能力不足 | 未集中收集登录和防火墙日志 | 异常活动难以及时发现 |

因此，现代内网安全强调：

> 不因为流量来自内网就自动信任，而是根据身份、设备、区域、业务需求和风险持续判断是否允许访问。

### 什么是网络分段

**网络分段（Network Segmentation）** 是根据业务功能和安全等级，将一个大的网络划分为多个区域，并限制区域之间的访问。

典型企业网络可以划分为：

| 安全区域 | 常见设备 | 基本策略 |
| --- | --- | --- |
| 互联网区 | 外部用户、外部服务 | 默认不可信 |
| DMZ 区 | 对外网站、邮件网关、反向代理 | 只开放必要服务，不直接访问核心系统 |
| 办公网 | 员工终端、打印机 | 不应直接管理服务器 |
| 服务器区 | IIS、数据库、文件服务器 | 仅允许业务流量和受控管理流量 |
| 管理区 | 堡垒机、运维终端 | 只允许授权管理员使用 |
| 身份与核心区 | 域控制器、证书服务、备份系统 | 最高保护级别，严格限制来源 |

网络分段的目标不是让不同区域“完全无法通信”，而是让通信满足三个条件：

1. 有明确业务需求。
2. 只开放必要的目标端口。
3. 可以记录和审计。

### 信任边界

**信任边界（Trust Boundary）** 是安全策略发生变化的位置。例如，办公网进入服务器区、服务器区进入域控区、互联网进入 DMZ 区，都是典型的信任边界。

跨越信任边界时，应回答以下问题：

| 问题 | 示例 |
| --- | --- |
| 谁发起访问？ | 普通办公终端、管理终端、应用服务器 |
| 访问哪个目标？ | IIS 服务器、数据库服务器、域控制器 |
| 使用什么协议？ | HTTPS、SMB、RDP、WinRM、LDAP |
| 是否确实需要？ | 办公网访问网站需要 HTTPS，但通常不需要 SMB |
| 如何记录？ | 防火墙日志、Windows 安全日志、堡垒机审计 |

### 本项目实验拓扑

本项目使用两个虚拟网络模拟边界实验网和内部服务器网：

```text
边界实验网：192.168.10.0/24

管理与验证终端              双网卡边界服务器
Kali                       SRV01
192.168.10.10      <---->  192.168.10.20
                            192.168.100.20
                                  |
                                  | 信任边界
                                  |
内部服务器网：192.168.100.0/24    |
                                  |
                            内部业务服务器
                            SRV02
                            192.168.100.30
```

| 虚拟机 | 角色 | 网卡配置 |
| --- | --- | --- |
| Kali Linux | 管理与验证终端 | `192.168.10.10/24`，连接边界实验网 |
| SRV01 | 双网卡边界服务器 | `192.168.10.20/24`，连接边界实验网；`192.168.100.20/24`，连接内部服务器网 |
| SRV02 | 内部业务服务器 | `192.168.100.30/24`，连接内部服务器网 |

此时：

| 通信路径 | 是否直接可达 | 原因 |
| --- | --- | --- |
| Kali -> SRV01 外网卡 | 是 | 位于同一网段 |
| SRV01 内网卡 -> SRV02 | 是 | 位于同一网段 |
| Kali -> SRV02 | 否 | Kali 没有到 `192.168.100.0/24` 的直接路径 |

> “网络不可达”是分段的基础，但不是完整的安全方案。后续仍需使用防火墙、身份认证和日志审计控制跨区域访问。

# 任务二 理解路由、NAT 与端口转发

## 理论知识

### 数据包如何到达目标

主机发送数据前，会先判断目标 IP 是否与自己位于同一网段。

```text
目标在同一网段
    |
    +-- 是 --> 通过 ARP 查询目标 MAC 地址 --> 直接发送
    |
    +-- 否 --> 将数据包发送给默认网关 --> 由路由器继续转发
```

例如，SRV02 的地址为 `192.168.100.30/24`：

| 目标地址 | 判断结果 | 下一步 |
| --- | --- | --- |
| `192.168.100.20` | 同一网段 | SRV02 直接发送给 SRV01 内网卡 |
| `192.168.10.10` | 不同网段 | SRV02 将数据包发送给默认网关 |

### 路由

**路由（Routing）** 是根据目标网络选择下一跳的过程。路由器负责在不同网络之间转发数据包。

一个简化的路由表可能如下：

| 目标网络 | 下一跳 | 含义 |
| --- | --- | --- |
| `192.168.100.0/24` | 直连 | 目标位于内部服务器网 |
| `192.168.10.0/24` | 直连 | 目标位于边界实验网 |
| `0.0.0.0/0` | 默认网关 | 其他流量交给默认网关处理 |

需要注意：

> 路由只决定“数据包往哪里走”，不代表数据包一定应该被允许。

是否允许访问，应由防火墙和访问控制策略决定。

### NAT

**NAT（Network Address Translation，网络地址转换）** 会修改数据包中的 IP 地址或端口号，并维护转换关系。

最常见的场景是多个内网主机共享一个出口地址访问外部网络。

```text
SRV02 发起请求
源地址：192.168.100.30:51000
目标地址：192.168.10.10:8000
        |
        v
SRV01 执行 NAT
源地址改写为：192.168.10.20:62001
目标地址保持：192.168.10.10:8000
        |
        v
Kali 返回响应
源地址：192.168.10.10:8000
目标地址：192.168.10.20:62001
        |
        v
SRV01 根据 NAT 转换表还原
目标地址改写为：192.168.100.30:51000
```

NAT 的关键是**状态表**。SRV01 必须记住转换前后的对应关系，返回流量才能交给正确的内网主机。

### 常见 NAT 类型

| 类型 | 说明 | 常见场景 |
| --- | --- | --- |
| SNAT | 修改源地址 | 内网主机访问外部网络 |
| DNAT | 修改目标地址 | 将外部访问转发给内网服务 |
| PAT / NAPT | 同时利用端口区分连接 | 多台内网主机共享一个出口地址 |

### 端口转发

**端口转发（Port Forwarding）** 是将一个入口地址和端口收到的流量交给另一个目标地址和端口。

例如：

```text
Kali 访问 192.168.10.20:8080
            |
            v
SRV01 将 TCP 连接代理到 192.168.100.30:80
            |
            v
SRV02 上的 IIS 返回测试页面
```

端口转发常用于发布内部服务或通过边界设备访问内部管理页面。但如果规则过宽，也会绕过原有隔离设计，让内部服务暴露到不必要的区域。

### `portproxy` 与完整 NAT 的区别

Windows Server 提供 `netsh interface portproxy`。它适合演示和少量 TCP 代理，但不能当作完整 NAT。

| 对比项 | `portproxy` | RRAS NAT |
| --- | --- | --- |
| 工作方式 | 由 Windows 接收一个 TCP 连接，再向目标建立新的 TCP 连接 | 在网络层执行路由和地址转换 |
| 协议范围 | TCP | 支持完整 NAT 场景 |
| 是否需要逐端口配置 | 是 | 不一定 |
| 适用场景 | 临时、明确、少量端口 | 需要完整路由和地址转换的服务器场景 |
| 能否替代防火墙 | 不能 | 不能 |

### NAT 为什么不能替代防火墙

NAT 和防火墙经常同时部署，但它们职责不同。

| 机制 | 主要职责 | 回答的问题 |
| --- | --- | --- |
| 路由 | 选择数据包路径 | 数据包往哪里走？ |
| NAT | 修改地址或端口 | 地址如何转换？ |
| 防火墙 | 按策略允许或拒绝流量 | 这次访问是否应该放行？ |
| 身份认证 | 判断使用者身份 | 发起访问的是谁？ |
| 日志审计 | 记录和分析行为 | 发生了什么？是否异常？ |

## 实践验证

### 实验2：观察 TCP 端口转发

先在 SRV02 安装 IIS，并创建测试页面：

```powershell
Install-WindowsFeature -Name Web-Server -IncludeManagementTools

Set-Content -Path "C:\inetpub\wwwroot\index.html" `
  -Value "<h1>SRV02 internal test page</h1>"
```

在 SRV01 上创建一条只允许 Kali 访问的实验规则：

```powershell
Start-Service iphlpsvc

netsh interface portproxy add v4tov4 `
  listenaddress=192.168.10.20 listenport=8080 `
  connectaddress=192.168.100.30 connectport=80 protocol=tcp

New-NetFirewallRule `
  -DisplayName "LAB-Allow-TCP-8080-from-Kali" `
  -Direction Inbound -Action Allow -Protocol TCP `
  -LocalAddress 192.168.10.20 -LocalPort 8080 `
  -RemoteAddress 192.168.10.10
```

在 Kali 上验证：

```bash
curl http://192.168.10.20:8080/
```

**观察重点：**

1. Kali 访问的是哪个地址？
2. IIS 实际运行在哪台主机？
3. 为什么新增转发规则后，Kali 仍然不等于获得了对整个内部服务器网的访问能力？

实验结束后删除临时规则：

```powershell
netsh interface portproxy delete v4tov4 `
  listenaddress=192.168.10.20 listenport=8080

Remove-NetFirewallRule -DisplayName "LAB-Allow-TCP-8080-from-Kali"
```

### 扩展阅读：RRAS

**RRAS（Routing and Remote Access Service）** 是 Windows Server 的路由和远程访问角色，可以提供路由、NAT 和 VPN 能力。

在本项目中，RRAS 的作用是帮助理解：

1. 双网卡服务器如何成为两个网段之间的路由节点。
2. 内部服务器如何通过默认网关访问其他网络。
3. NAT 如何维护转换状态。

RRAS 配置容易改变实验网络的可达性，建议由教师演示或在虚拟机快照中单独完成，不与 `portproxy` 实验混用。

# 任务三 理解隧道、代理与横向移动风险

## 理论知识

### 什么是隧道

**隧道（Tunneling）** 是将一种网络流量封装在另一条连接中传输。

隧道本身并非恶意技术。管理员可以使用 SSH 隧道访问内部管理页面，企业也可能通过 VPN 建立安全远程接入。但如果未经审批的隧道绕过了原有边界策略，就会产生安全风险。

```text
正常访问路径：
管理终端 --> 防火墙策略检查 --> 内部服务

隧道访问路径：
管理终端 --> 隧道入口 ===== 封装连接 ===== 隧道出口 --> 内部服务
```

### 正向代理与反向连接

理解内网穿透的关键，是区分谁主动建立连接。

| 模式 | 连接方向 | 典型用途 |
| --- | --- | --- |
| 正向代理 | 客户端主动连接代理，再访问目标 | 统一上网出口、内容过滤 |
| 反向连接 | 内网客户端主动连接外部服务端，保持控制通道 | 远程运维、临时服务发布 |

很多企业防火墙对入站访问限制较严，但对服务器出站连接限制较弱。此时，内网主机如果主动连接外部服务，就可能建立一条绕过原有入口控制的通道。

```text
原本：
外部终端 --X--> 内部服务器

建立反向连接后：
内部服务器 ----主动连接----> 外部服务端
外部终端 ------访问映射端口------> 外部服务端
外部服务端 =====通过既有连接=====> 内部服务器
```

这说明：

> 内网安全不仅要控制“谁能进入”，还要控制“服务器能够主动连接到哪里”。

### SSH 本地端口转发

SSH 本地端口转发会在本地监听一个端口，并通过 SSH 会话让远端主机访问指定目标。

```text
Kali 本地端口 18080
        |
        v
SSH 加密会话到 SRV01
        |
        v
SRV01 访问 SRV02:80
```

它适合通过堡垒机访问内部 Web 管理页。与开放整个内网相比，单端口转发更容易限制和审计。

### frp 的基本架构

frp 是一种反向代理工具，由服务端 `frps` 和客户端 `frpc` 组成。

```text
外部服务端 frps
192.168.10.10:7000
        ^
        |
        | frpc 主动建立控制连接
        |
内部客户端 frpc
SRV02
        |
        +--> 本地 IIS 127.0.0.1:80

访问者连接 frps 的映射端口
        |
        v
frps 通过既有通道将流量交给 frpc
        |
        v
frpc 将流量发送给本地 IIS
```

从防御视角看，应关注：

| 控制点 | 防御措施 |
| --- | --- |
| 工具运行 | 应用白名单、EDR、进程审计 |
| 主动外连 | 出站防火墙、代理日志、边界流量分析 |
| 固定端口 | 监控非常用出站端口，但不能只依赖端口号 |
| 访问行为 | 识别服务器异常持续连接、周期性心跳和未知目的地址 |
| 合法使用 | 审批、资产登记、固定版本、强认证、限定端口 |

### 代理与 SOCKS

代理代表客户端访问其他目标。SOCKS5 是一种通用代理协议，可以承载多种 TCP 访问需求。

单个端口转发与 SOCKS 代理的差异如下：

| 类型 | 访问范围 | 风险 |
| --- | --- | --- |
| 单端口转发 | 仅访问指定目标和端口 | 较容易限制和审计 |
| SOCKS 代理 | 通过代理访问多个目标 | 访问能力更强，滥用风险更高 |

基础课程只演示单端口转发，不部署通用 SOCKS 代理。

### 横向移动

**横向移动（Lateral Movement）** 是攻击者进入一台主机后，继续访问同一网络中的其他主机，扩大控制范围的过程。

横向移动通常需要同时满足多个条件：

```text
网络可达
   +
存在可访问的远程服务
   +
获得可用凭据或利用条件
   +
目标账号具备足够权限
   =
攻击可能扩散
```

| 条件 | 常见问题 | 防御措施 |
| --- | --- | --- |
| 网络可达 | 办公网可直接访问服务器管理端口 | 网络分段、主机防火墙 |
| 服务暴露 | RDP、SMB、WinRM 向大范围开放 | 关闭非必要服务、限制来源地址 |
| 凭据可用 | 弱口令、重复密码、高权限账号混用 | 强密码、MFA、账号分离、凭据保护 |
| 权限过大 | 普通账号拥有本地管理员权限 | 最小权限、特权账号管理 |
| 缺少检测 | 大量失败登录无人关注 | 日志集中采集、异常行为告警 |

# 任务四 识别内网暴露面与异常行为

## 理论知识

### 什么是暴露面

**暴露面（Attack Surface）** 是系统中可能被访问、调用或利用的入口集合。

内网服务器的暴露面不仅包括开放端口，还包括账号、共享目录、远程管理方式、信任关系和出站连接。

| 维度 | 需要回答的问题 | 常见工具或日志 |
| --- | --- | --- |
| 资产 | 网络中有哪些主机？分别承担什么业务？ | 资产台账、CMDB |
| 地址与路由 | 主机连接了哪些网络？默认网关是谁？ | `ipconfig /all`、`route print` |
| 端口与服务 | 哪些服务正在监听？是否确实需要？ | `netstat -ano`、`Get-NetTCPConnection` |
| 账号与权限 | 哪些账号具有管理员权限？是否混用？ | `whoami /all`、本地组、AD 组 |
| 网络连接 | 主机正在连接哪些地址？是否异常？ | 防火墙日志、EDR、网络流量日志 |
| 登录行为 | 是否存在大量失败登录或异常时间登录？ | Windows 安全日志 |

### 端口、服务与风险

端口只是网络入口，真正需要评估的是端口背后的服务、身份验证方式和访问范围。

| 服务 | 常见端口 | 合法用途 | 常见风险 | 基本控制 |
| --- | --- | --- | --- | --- |
| HTTP / HTTPS | `80`、`443` | 网站和 API | Web 漏洞、错误暴露后台 | 反向代理、TLS、应用加固 |
| SMB | `445` | 文件共享、部分远程管理 | 共享越权、凭据中继、横向移动 | 限定来源、SMB 签名 |
| RDP | `3389` | 图形化远程管理 | 弱口令、暴力尝试、账号滥用 | 管理区访问、MFA、审计 |
| WinRM | `5985`、`5986` | PowerShell 远程管理 | 管理接口暴露、凭据滥用 | 限定来源、优先 HTTPS、审计 |
| LDAP / LDAPS | `389`、`636` | 目录服务查询 | 中继、明文或弱保护查询 | 签名、通道绑定、TLS |

### 为什么要关注出站连接

传统运维经常重点检查入站端口，却忽略服务器主动访问外部网络的行为。

服务器出站连接可能包括：

| 类型 | 示例 | 处置思路 |
| --- | --- | --- |
| 正常业务 | 更新源、时间同步、备份、调用 API | 建立白名单并记录 |
| 配置错误 | 连接废弃地址、错误代理 | 修正配置 |
| 未审批工具 | 私自部署远程控制、代理或隧道 | 调查来源并隔离 |
| 恶意活动 | 周期性回连、访问未知公网地址 | 立即响应并开展取证 |

### Windows 安全日志

日志是发现异常行为的重要依据。

| 事件 ID | 含义 | 关注点 |
| --- | --- | --- |
| `4624` | 登录成功 | 登录类型、来源地址、账号、时间 |
| `4625` | 登录失败 | 是否短时间内大量失败 |
| `4720` | 创建用户账号 | 是否存在未经审批的新账号 |
| `4732` | 将成员加入本地安全组 | 是否将普通账号加入管理员组 |
| `4776` | NTLM 凭据验证 | 是否存在不必要的 NTLM 使用 |

日志不能只保存在本机。主机一旦失陷，本地日志可能被删除或篡改，因此生产环境应集中收集到 SIEM 或日志平台。

# 任务五 构建内网纵深防御体系

## 理论知识

### 为什么需要纵深防御

单一安全措施总有边界：

| 单一措施 | 局限 |
| --- | --- |
| 只部署边界防火墙 | 无法阻止已经进入内网的攻击扩散 |
| 只使用杀毒软件 | 无法替代账号治理、网络隔离和日志审计 |
| 只启用强密码 | 无法解决凭据泄露和权限过大问题 |
| 只配置 NAT | NAT 不是访问控制策略 |
| 只做日志记录 | 没有告警和响应流程，日志难以及时发挥作用 |

**纵深防御（Defense in Depth）** 是在多个层次设置互补控制。即使某一层失效，其他层仍可限制攻击范围，并为检测和响应争取时间。

### 内网纵深防御模型

```text
第一层：网络分段
    |
    +-- 办公网、管理区、服务器区、域控区分离
    +-- 默认拒绝跨区域访问
    +-- 只开放必要端口

第二层：身份与权限
    |
    +-- 普通账号与管理账号分离
    +-- 高权限账号只在专用管理路径使用
    +-- 最小权限、MFA、凭据保护

第三层：协议与主机加固
    |
    +-- 启用主机防火墙
    +-- 启用 SMB 签名
    +-- 减少 NTLM 使用
    +-- 关闭非必要服务

第四层：出站控制与检测
    |
    +-- 限制服务器访问未知外部地址
    +-- 发现异常持续连接和未知工具
    +-- 集中采集日志

第五层：响应与恢复
    |
    +-- 快速隔离
    +-- 凭据轮换
    +-- 取证分析
    +-- 备份恢复与复盘
```

### 最小暴露面原则

最小权限原则不仅适用于用户权限，也适用于网络访问。

错误做法：

```text
办公网所有主机 --> 服务器区所有端口
```

更合理的做法：

```text
普通终端 --> IIS:443
运维终端 --> 堡垒机
堡垒机 --> 授权服务器:RDP 或 WinRM
应用服务器 --> 数据库服务器:业务端口
授权管理路径 --> 域控制器:必要管理端口
其他流量 --> 默认拒绝
```

每增加一条放行规则，都应明确：

1. 来源区域或来源主机。
2. 目标主机。
3. 目标端口。
4. 业务负责人。
5. 有效期。
6. 日志与复核方式。

### SMB 签名

SMB 签名会为 SMB 消息提供完整性保护。接收方可以验证消息是否被篡改，并降低部分 SMB 中继风险。

| 状态 | 含义 |
| --- | --- |
| 支持签名 | 双方可以在协商后使用签名 |
| 要求签名 | 未签名的 SMB 通信会被拒绝 |

Windows Server 2025 默认要求入站和出站 SMB 签名。较早系统仍应显式检查和配置，同时评估旧版客户端、第三方 NAS 和业务程序兼容性。

### `Protected Users`

`Protected Users` 是 Active Directory 域中的安全组，适用于需要额外保护的域账号。加入该组后，域控制器会限制 NTLM、摘要式身份验证和凭据委派等较弱机制。

它不是“勾选后一定更安全且没有副作用”的开关：

| 优点 | 可能影响 |
| --- | --- |
| 降低高权限凭据被重复利用的风险 | 旧系统或依赖 NTLM 的应用可能无法登录 |
| 减少凭据缓存和委派风险 | 部分计划任务、远程管理流程需要调整 |
| 推动使用 Kerberos | 非域环境不适用 |

因此，应先使用测试账号验证兼容性，再逐步应用到高权限账号。

### 特权访问模型

传统 Tier 0 / Tier 1 / Tier 2 模型有助于理解账号隔离。Microsoft 当前更推荐企业访问模型（Enterprise access model），但核心原则一致：

> 高权限账号不应出现在低信任设备上。

| 账号类型 | 允许登录位置 | 禁止事项 |
| --- | --- | --- |
| 域和身份系统管理账号 | 专用安全管理工作站、域控管理路径 | 不登录普通办公终端和业务服务器 |
| 服务器管理账号 | 专用管理节点、授权服务器 | 不作为日常办公账号 |
| 普通办公账号 | 普通终端和业务系统 | 不授予服务器管理权限 |

### `krbtgt` 密码重置

`krbtgt` 是 Active Directory 中用于 Kerberos 票据保护的重要账号。当域已经遭受严重入侵、需要使伪造票据失效时，才考虑重置其密码。

这不是日常实验操作，也不能简单理解为“连续执行两条改密命令”：

1. 先确认域控制器复制健康。
2. 使用 Microsoft 提供的脚本或组织批准的流程。
3. 第一次重置后，等待复制和票据生命周期窗口。
4. 再进行第二次重置。
5. 在维护窗口内完成业务验证和应急准备。

官方脚本：<https://github.com/microsoft/New-KrbtgtKeys.ps1>

### 防御措施与风险对应关系

| 风险 | 优先防御措施 | 原因 |
| --- | --- | --- |
| 办公网直接访问服务器管理端口 | 网络分段、主机防火墙 | 降低横向移动可达性 |
| 弱口令或账号混用 | 强密码、MFA、账号分离 | 降低凭据滥用风险 |
| SMB 中继和篡改 | SMB 签名、减少 NTLM | 加强协议保护 |
| 未审批隧道和代理 | 出站控制、EDR、流量分析 | 发现绕过边界的通道 |
| 域管理员凭据泄露 | 专用管理终端、特权访问模型 | 防止高权限凭据落入低信任设备 |
| 失陷后难以发现 | 集中日志、SIEM、告警 | 缩短发现和响应时间 |

### 实验5：检查主机防火墙和 SMB 签名

在 SRV02 上执行：

New-NetFirewallRule `

```powershell
Remove-NetFirewallRule -DisplayName "LAB-Allow-RDP-from-AdminHost"
```

3. 为什么启用 SMB 签名前仍需进行兼容性测试？

# 项目八知识点总结

| 知识点 | 核心结论 |
| --- | --- |
| 内网安全 | 内网不是天然可信区域，需要持续控制和审计 |
| 网络分段 | 根据业务和安全等级划分区域，限制跨区域访问 |
| 信任边界 | 安全策略发生变化的位置，跨越边界必须有明确规则 |
| 路由 | 决定数据包往哪里走 |
| NAT | 修改地址或端口并维护转换状态，不能替代防火墙 |
| 端口转发 | 将指定入口流量交给内部服务，应坚持最小暴露面 |
| 隧道 | 可以用于合法运维，也可能绕过入口控制 |
| 横向移动 | 通常依赖网络可达、服务暴露、凭据可用和权限过大 |
| 暴露面盘点 | 同时关注资产、端口、服务、身份、连接和日志 |
| 纵深防御 | 网络、身份、主机、协议、审计和响应必须互相补充 |
