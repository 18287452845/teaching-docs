# 实验二：用户账户暴力破解与凭据提取渗透 · 课堂逐字稿

---

## 开场导入（3 分钟）

同学们好，今天我们做实验二。这个实验的主题是"暴力破解与凭据提取"。

先问大家一个问题：如果你是一个攻击者，面对一台 Windows 服务器，你手里什么凭据都没有——没有用户名、没有密码，你会怎么一步步拿下这台机器？

这就是今天实验要走的完整攻击链。我们会从零开始，经历五个阶段：先搞清楚目标上有哪些账户，再尝试破解密码，拿到管理员权限后从内存里提取更多凭据，检查有没有隐藏后门，最后站在防御方的角度把这些漏洞全部堵上。

在动手之前，我们需要先理解三个前置知识点。这三个知识点不是孤立的，它们分别对应攻击链的不同阶段。我画一张对照表给大家看：

- 第一个知识点"密码哈希与 NTLM 认证"，对应阶段一和阶段二——你得先知道密码在系统里长什么样，才能理解为什么弱口令能被快速破解。
- 第二个知识点"SAM 数据库与 LSASS 进程"，对应阶段三——你得知道凭据存在哪里，才能理解 Mimikatz 为什么能一次性把所有密码都读出来。
- 第三个知识点"隐藏账户"，对应阶段四——攻击者拿到权限后怎么留后门，管理员又怎么发现它。

好，我们从第一个知识点开始。

## 第一部分：前置知识讲解（30 分钟）

### 知识点一：密码哈希与 NTLM 认证（15 分钟）

#### 1.1 Windows 密码存储方式（5 分钟）

第一个问题，也是最基础的问题：Windows 到底保存的是什么？

答案是：现代 Windows 不保存明文密码，保存的是密码的哈希值。

什么是哈希？简单说就是一个单向函数——你给它一个密码，它输出一串固定长度的乱码；但你拿着这串乱码，没办法反推出原始密码。这就是"单向"的意思。

Windows 把这些哈希值存在一个叫 SAM 的数据库里，全称 Security Account Manager，路径在 `C:\Windows\System32\config\SAM`。每个本地用户在 SAM 里都有一条记录，核心就是用户名和对应的 NTLM Hash。

这里有个历史遗留问题。Windows 早期用的是 LM Hash，这个算法非常弱——它会把密码转成大写、截断成两个 7 字符的段分别加密。所以即使你设了一个 14 位密码，攻击者实际上只需要分别破解两个 7 位的段，难度大大降低。从 Vista 和 Server 2008 开始，LM Hash 默认禁用了，现在用的是 NTLM Hash，基于 MD4 算法，保留大小写，支持长密码。

但 NTLM Hash 也有一个致命缺陷：**没有盐值**。什么意思？就是同一个密码，不管在哪台机器上、哪个用户设的，算出来的哈希值永远一样。"123456"在你的机器上是一个固定的哈希，在我的机器上也是同一个哈希。这就意味着攻击者可以提前算好一张"密码→哈希"的对照表——也就是彩虹表——拿到哈希后直接查表就能还原密码。

记住这两个关键点，后面实验会反复用到：
1. 弱密码的哈希可以被彩虹表秒破
2. NTLM Hash 本身就能用来认证（这个马上讲）

#### 1.2 NTLM 认证流程（10 分钟）

好，现在我们知道密码以哈希形式存在 SAM 里。下一个问题：当你登录一台远程服务器，或者访问一个网络共享时，系统怎么验证你的身份？

Windows 用的是 NTLM 协议，全称 NT LAN Manager。它的核心机制叫"挑战/响应"。我用一个比喻来解释：

想象你去一个需要暗号的地下俱乐部。门卫不会让你直接喊出暗号——万一旁边有人偷听呢？所以门卫的做法是：他给你出一道数学题，这道题只有知道暗号的人才能算对。你算出答案交给他，他用自己知道的暗号验算一遍，对了就放你进去。

NTLM 认证就是这个过程，分四步：

第一步，协商。客户端告诉服务器："我是 weakadmin，我要登录。"

第二步，挑战。服务器生成一个 16 字节的随机数发回来，这就是"题目"。每次都不一样，所以你不能背答案。

第三步，响应。客户端用自己内存中的 NTLM Hash 对这个随机数做一次加密运算，把结果发回去。注意，发的不是哈希本身，而是哈希加密随机数后的结果。

第四步，验证。服务器从自己的 SAM 里取出同一个用户的 NTLM Hash，用同样的方法算一遍。如果结果和客户端发来的一样，说明对方确实知道正确的密码——认证通过。

这个设计的初衷是好的：明文密码从不出网，哈希也不直接传输，每次挑战随机不同所以不能重放。听起来很安全对吧？

但问题出在第三步。客户端计算响应时，唯一需要的材料是什么？是 NTLM Hash 本身。它不需要知道原始密码是什么，只要有哈希就能完成认证。

这就引出了两个攻击面：

第一个，暴力破解。如果服务器没有设置账户锁定策略，攻击者可以无限次地触发这个四步流程，每次用不同的密码试。试对了就进去了。这是我们阶段二要做的事。

第二个，Pass-the-Hash，哈希传递攻击。如果攻击者从内存中偷到了哈希，他根本不需要破解出明文密码，直接拿哈希就能完成第三步的计算，服务器照样会放行。这是阶段三 Mimikatz 之后我们要演示的。

所以你看，NTLM 的安全性完全依赖于两个前提：一是密码足够强不会被爆破，二是哈希不会泄露。一旦这两个前提被打破，整个认证机制就形同虚设。

关于 Kerberos，这里简单提一句。域环境下 Windows 默认用 Kerberos 而不是 NTLM，它基于票据机制，安全性更高。但我们今天的实验是工作组环境，用的就是 NTLM，所以 Kerberos 大家了解即可，不考核。

### 知识点二：SAM 数据库与 LSASS 进程（10 分钟）

好，第一个知识点解决了"密码以什么形式存在、认证怎么用它"。第二个知识点回答下一个问题：这些哈希和凭据具体存放在哪里？攻击者拿到管理员权限后能读到什么？

答案是两个位置：磁盘上的 SAM，和内存中的 LSASS。

#### 2.1 SAM 数据库（3 分钟）

SAM 我们刚才已经提过了。它是一个文件，路径在 `C:\Windows\System32\config\SAM`。所有本地用户的用户名、NTLM Hash、RID、最后登录时间等信息都记录在这里。

SAM 有两个重要特性：
- 第一，系统运行时 SAM 文件被锁定，普通方式无法直接复制。但如果你有 SYSTEM 权限，可以通过注册表或者 Mimikatz 的 `lsadump::sam` 命令读取。
- 第二，SAM 是静态的——只有创建账户或修改密码时才会更新。它不会记录"当前谁在线"。

所以 SAM 能告诉攻击者"这台机器上有哪些账户、它们的哈希是什么"，但不能告诉他"现在谁登录着、他们的明文密码是什么"。要回答后面这个问题，就需要看 LSASS。

#### 2.2 LSASS 进程（5 分钟）

LSASS，全称 Local Security Authority Subsystem Service，进程名是 lsass.exe。它是 Windows 安全认证的核心进程。

每当一个用户登录系统——不管是本地登录、远程桌面还是网络访问——LSASS 都会把这个用户的凭据缓存到内存中。缓存什么呢？NTLM Hash、Kerberos 票据，甚至在某些配置下还有明文密码。

为什么要缓存？因为 Windows 需要用这些凭据去访问网络资源。比如你登录了一台服务器，然后要访问另一台机器的共享文件夹，系统需要用你的凭据去做 NTLM 认证。如果每次都让你重新输密码，体验就太差了。所以 LSASS 把凭据留在内存里，需要时直接用。

这就带来一个严重的安全问题：如果一台服务器上有多个用户登录过——比如管理员远程桌面上去维护过、某个服务账户在运行——那 LSASS 内存里就同时存着所有这些账户的凭据。攻击者只要有管理员权限，用 Mimikatz 读一次 LSASS 内存，就能把所有人的凭据一锅端。

这就是为什么阶段三不会止步于"用 weakadmin 登录系统"。登录只是手段，目的是读取 LSASS，从一个账户扩展到所有账户。

那怎么防御？三层防护：
1. **RunAsPPL**：让 LSASS 以受保护进程运行，禁止调试 API 读取它的内存。这是最直接的防御。
2. **Credential Guard**：用虚拟化技术把凭据隔离到一个独立的安全区域，即使攻击者有 SYSTEM 权限也读不到。但这需要硬件支持 TPM 2.0。
3. **Protected Users 组**：把用户加入这个安全组后，系统不会为该用户缓存明文密码和长期凭据。

这三层防护我们在阶段五会实际配置和验证。

#### 2.3 明文密码缓存：WDigest（2 分钟）

补充一个细节。刚才说 LSASS 里"某些配置下"会有明文密码，这个配置就是 WDigest。

WDigest 是一个老的认证协议，它要求系统在内存中保留明文密码。Windows 10 1607 和 Server 2016 之后默认禁用了 WDigest，所以正常情况下 LSASS 里只有哈希没有明文。

但在我们的实验环境中，初始化脚本已经手动启用了 WDigest（`UseLogonCredential=1`），这样 Mimikatz 才能演示提取明文密码。实际生产环境中，这个选项应该始终保持禁用。

### 知识点三：持久化手段——隐藏账户（5 分钟）

好，最后一个知识点。前面两个知识点讲的是攻击者怎么"进来"和"拿东西"。这个知识点讲的是攻击者怎么"留下来"。

现实中的攻击不是一次性的。攻击者拿到权限后，第一件事往往不是搞破坏，而是给自己留一个后门——万一管理员改了密码、清了会话，他还能回来。

最简单的本地持久化手段之一，就是创建一个隐藏账户。Windows 有一个机制：如果用户名以美元符号 `$` 结尾，这个账户就不会出现在 `net user` 的列表里，也不会显示在登录界面上。

比如攻击者执行 `net user backdoor$ P@ssw0rd123 /add`，再把它加到管理员组。之后管理员日常巡检时敲 `net user`，看不到这个账户；看登录界面，也看不到。但这个账户是真实存在的，攻击者随时可以用它登录。

那怎么发现它？关键在于：`$` 后缀只是骗过了 `net user` 这个命令的显示逻辑，账户本身在注册表 SAM 中是完整记录的。所以用 `Get-LocalUser`、`wmic useraccount`、或者直接查注册表，都能看到它。

这就是为什么我们在阶段四要用多种工具交叉检测——单靠 `net user` 是不够的。

好，三个前置知识点讲完了。我们快速回顾一下：
1. 密码以无盐哈希存储，弱密码可被彩虹表秒破；NTLM 认证中哈希本身就能充当密码
2. SAM 存静态哈希，LSASS 缓存动态凭据；管理员权限可读取 LSASS 内存
3. `$` 后缀账户绕过常用管理命令，但注册表中仍有记录

接下来我们进入实验操作部分。

## 第二部分：实验环境配置（15 分钟）

### 环境概述（2 分钟）

我们的实验环境很简单：两台虚拟机，一台 Kali Linux 作为攻击机，一台 Windows Server 2025 作为靶机，两台机器在同一个 NAT 网段 192.168.1.0/24。

Kali 的 IP 是 192.168.1.10，Windows Server 是 192.168.1.20。

### 靶机配置讲解（13 分钟）

靶机配置分三步，必须按顺序来。我解释一下为什么要这个顺序：

**第一步：放开密码策略。** 我们要创建弱密码账户（比如密码是 123456），但 Windows Server 2025 默认要求密码复杂度。如果不先关掉复杂度要求，后面创建弱密码账户的命令会失败。所以第一步是打开 `secpol.msc`，把密码复杂度禁用、最小长度设为 0、账户锁定阈值设为 0。

**第二步：放开空会话限制。** Windows Server 2025 默认禁止匿名用户枚举 SAM 账户。但我们阶段一要演示的就是"攻击者在没有任何凭据的情况下枚举用户列表"，所以必须手动放开这些限制。这一步涉及 secpol.msc 里的三个安全选项，加上几条注册表命令。

**第三步：运行初始化脚本。** 这个 PowerShell 脚本会创建弱密码测试账户、创建隐藏账户、启用远程桌面、关闭防火墙、启用 SMB1、关闭 Defender、启用 WDigest 等等。所有这些都是为了构造一个"漏洞百出"的靶机环境，让我们的攻击链能够顺利走通。

现在大家按照讲义上的步骤配置靶机。配置完成后重启，然后在 Kali 上跑两条验证命令：

```
smbclient -L //192.168.1.20 -N
rpcclient -U "" 192.168.1.20 -c "enumdomusers"
```

第一条应该返回共享列表，第二条应该返回用户列表。如果都正常，说明环境就绪，我们可以开始攻击了。

大家配置的时候如果遇到问题举手，我来看。

## 第三部分：实验操作（60 分钟）

### 阶段一：SMB 空会话枚举（10 分钟）

好，环境都就绪了。我们现在站在攻击者的视角，从零开始。

此刻我们只知道目标 IP 是 192.168.1.20，其他什么都不知道。第一步要做的是信息收集——搞清楚这台机器上有哪些用户账户。

为什么能做到？因为 Windows 的 SMB 协议在某些配置下允许"空会话"连接——也就是不提供任何用户名和密码，匿名连接上去，然后查询系统信息。这在早期 Windows 中是默认允许的，现代系统虽然默认关闭了，但很多管理员不知道要检查这个配置。

我们用三个工具来枚举：

第一个，`enum4linux -a 192.168.1.20`。这是一个综合枚举工具，会尝试获取用户列表、共享资源、密码策略等所有能拿到的信息。大家运行一下，看输出。

第二个，`rpcclient -U "" 192.168.1.20 -c "enumdomusers"`。这个更精准，直接通过 RPC 调用枚举域/本地用户。`-U ""` 表示空用户名，也就是匿名连接。

第三个，`nxc smb 192.168.1.20 --users`。NetExec 是 CrackMapExec 的继任者，功能类似但更现代。

大家看输出，应该能看到 user1、user2、user3、user4、weakadmin、Administrator 这些账户名。

然后我们再查一下密码策略：`nxc smb 192.168.1.20 --pass-pol`。看输出——密码最小长度 0，锁定阈值 None。这意味着什么？意味着我们可以无限次尝试密码，不会被锁定。

好，阶段一完成。我们现在手里有了用户名列表和密码策略信息。下一步就是把这些用户名变成可登录的凭据。

### 阶段二：密码暴力破解（15 分钟）

现在我们手里有什么？用户名列表：user1、user2、user3、user4、weakadmin、administrator。密码策略：无复杂度要求、无锁定。

这就是暴力破解的完美条件。没有锁定意味着我们可以无限次尝试，没有复杂度要求意味着用户很可能设了弱密码。

我们用 Hydra 来做 SMB 暴力破解。Hydra 是一个在线密码破解工具，它会自动对目标发起认证请求，逐一尝试用户名和密码的组合。

首先准备两个字典文件。用户名字典就是我们刚才枚举到的那些账户名，密码字典用常见弱密码——123456、password、admin123 这些。

```
echo -e "user1\nuser2\nuser3\nuser4\nweakadmin\nadministrator" > /tmp/users.txt
echo -e "123456\npassword\nadmin\nadmin123\nP@ssw0rd\n123456789\nqwerty" > /tmp/passwords.txt
```

然后运行 Hydra：

```
hydra -L /tmp/users.txt -P /tmp/passwords.txt smb://192.168.1.20
```

`-L` 指定用户名字典，`-P` 指定密码字典，`smb://` 表示走 SMB 协议。

大家看输出。Hydra 会显示成功破解的账户，应该能看到 user1/123456、user2/password、weakadmin/admin123 这些结果。

这里我要强调一点：为什么这能成功？回到前置知识——NTLM 认证的每次尝试都是一个完整的四步流程（协商→挑战→响应→验证）。如果服务器设了账户锁定策略，比如"5 次失败后锁定 15 分钟"，那 Hydra 试到第 5 次就会被挡住。但我们的靶机锁定阈值是 0，也就是永不锁定，所以 Hydra 可以一直试下去直到找到正确密码。

接下来我们再演示一下密码喷射。密码喷射和暴力破解的区别是什么？暴力破解是"一个用户试很多密码"，密码喷射是"一个密码试很多用户"。在有锁定策略的环境中，密码喷射更隐蔽，因为每个用户只被尝试一次。

```
nxc smb 192.168.1.20 -u /tmp/users.txt -p '123456'
```

看输出，带 `[+]` 标记的就是认证成功的。

最后我们验证一下破解到的管理员凭据 weakadmin/admin123 是否真的能登录：

```
xfreerdp3 /v:192.168.1.20 /u:weakadmin /p:admin123 /cert:tofu
```

如果远程桌面窗口弹出来了，说明我们已经拿到了这台服务器的管理员权限。

好，阶段二完成。我们从"知道有哪些用户"升级到了"拥有一个管理员账户的密码"。但这只是一个入口。下一步，我们要利用这个管理员权限，从内存中提取更多凭据。

### 阶段三：Mimikatz 凭据提取（20 分钟）

这是今天实验最核心的阶段。

我们现在有了 weakadmin/admin123 这个本地管理员账户。如果我们只是用它来登录、做做日常操作，那攻击到这里就结束了。但真正的攻击者不会满足于此——他要的是这台机器上所有人的凭据。

为什么？因为服务器上可能有域管理员登录过、有服务账户在运行、有其他管理员远程维护过。这些人的凭据都缓存在 LSASS 内存里。拿到这些凭据，攻击者就能横向移动到其他机器。

工具就是 Mimikatz。它是安全研究员 Benjamin Delpy 开发的，本质上是一个 Windows 凭据提取工具。

我们先通过远程桌面登录到靶机，以管理员身份打开命令行，进入 Mimikatz 所在目录：

```
cd C:\Tools
mimikatz.exe
```

进入 Mimikatz 交互界面后，第一步是获取调试权限：

```
privilege::debug
```

这条命令请求 SeDebugPrivilege 权限。为什么需要它？因为 LSASS 是一个系统进程，普通程序不能读取其他进程的内存。但拥有 SeDebugPrivilege 的程序可以——这个权限本来是给调试器用的，Mimikatz 借用了这个机制。

如果输出 `Privilege '20' OK`，说明成功了。

第二步，提升到 SYSTEM 令牌：

```
token::elevate
```

这一步把当前进程的安全上下文提升到 NT AUTHORITY\SYSTEM，这是 Windows 中最高权限的身份。

第三步，导出 SAM 数据库中的哈希：

```
lsadump::sam
```

这条命令从磁盘上的 SAM 文件读取所有本地账户的 NTLM Hash。大家看输出，每个用户都有一个 `Hash NTLM` 字段，那就是他们的密码哈希。

第四步，从 LSASS 内存中提取凭据：

```
sekurlsa::logonpasswords
```

这是 Mimikatz 最强大的命令。它直接读取 LSASS 进程的内存，导出所有已登录会话的凭据。如果 WDigest 启用了并且用户重新登录过，你会看到明文密码；否则至少能看到 NTLM Hash。

注意：如果在 Server 2025 上遇到报错 `ERROR kuhl_m_sekurlsa_acquireLSA`，说明 Mimikatz 版本不完全兼容 Server 2025 的 LSASS 结构。这时候我们有替代方案——从 Kali 端用 Impacket 远程导出：

```
impacket-secretsdump weakadmin:admin123@192.168.1.20
```

效果一样，而且不需要在靶机上运行任何工具。

#### Pass-the-Hash 演示

拿到哈希后，我们来演示 Pass-the-Hash。回到 Kali 攻击机：

```
impacket-psexec -hashes :<NTLM哈希> weakadmin@192.168.1.20
```

注意看：我们没有输入任何明文密码，只用了哈希值，就拿到了一个 SYSTEM 权限的 shell。这就是 Pass-the-Hash 的威力——哈希等于密码。

为什么能成功？回到前置知识 1.2：NTLM 认证的第三步，客户端用哈希加密挑战。Impacket 做的事情就是：拿着我们提供的哈希，直接完成第三步的计算，服务器无法区分这是"用户输入了正确密码后系统算出的响应"还是"攻击者直接用偷来的哈希算出的响应"。

好，阶段三完成。我们从一个管理员密码扩展到了整台机器上所有账户的凭据。接下来要检查：攻击者有没有留下后门？

### 阶段四：隐藏账户检测（8 分钟）

现在我们切换视角——从攻击者变成防御方。

假设你是这台服务器的管理员，你怀疑系统被入侵过。你要检查有没有被留下后门账户。

最常用的检查命令是什么？`net user`。大家在靶机上敲一下：

```
net user
```

看输出，有 user1、user2、user3、user4、weakadmin、Administrator、Guest。看起来没什么异常对吧？

但实际上我们的初始化脚本创建了一个 `backdoor$` 账户。它去哪了？

这就是 `$` 后缀的隐藏效果——`net user` 命令的显示逻辑会跳过以 `$` 结尾的账户名。注意，不是账户不存在，只是这个命令不显示它。

那怎么发现它？我们用其他工具：

方法一，PowerShell 的 `Get-LocalUser`：

```powershell
Get-LocalUser | Select-Object Name, Enabled, Description
```

看，`backdoor$` 出现了。因为 `Get-LocalUser` 是直接从 SAM 注册表读取的，不受 `$` 后缀的显示过滤影响。

方法二，直接查注册表：

```
reg query "HKLM\SAM\SAM\Domains\Account\Users\Names"
```

这里列出的是 SAM 中所有用户的注册表键，`backdoor$` 也在里面。

方法三，从 Kali 远程检测：

```
nxc smb 192.168.1.20 -u weakadmin -p 'admin123' --users
```

NetExec 会列出所有用户，包括隐藏的。

所以结论是什么？`net user` 看不到不等于账户不存在。日常安全巡检中，应该用 `Get-LocalUser` 或者定期审计 SAM 注册表，而不是只依赖 `net user`。

好，阶段四完成。我们已经走完了完整的攻击链：枚举→爆破→提取凭据→发现后门。最后一个阶段，我们要把这些漏洞全部堵上。

### 阶段五：密码策略加固与验证（12 分钟）

前四个阶段暴露了四类问题，现在我们逐一加固：

**第一，密码策略加固。**

打开 `secpol.msc`，或者用命令行：

```powershell
net accounts /minpwlen:8 /maxpwage:90 /uniquepw:5
net accounts /lockoutthreshold:5 /lockoutduration:15 /lockoutwindow:15
gpupdate /force
```

这做了什么？密码最少 8 位、90 天必须换、记住最近 5 个密码不能重复、5 次失败锁定 15 分钟。

验证一下：`net accounts`，看输出是否生效。

然后我们回到 Kali 再跑一次 Hydra：

```
hydra -l user1 -P /tmp/passwords.txt smb://192.168.1.20 -t 1 -w 5
```

这次你会发现，试了 5 次之后就不行了——账户被锁定了。这就是账户锁定策略的效果。

**第二，LSA 保护，防御 Mimikatz。**

```powershell
reg add "HKLM\SYSTEM\CurrentControlSet\Control\Lsa" /v RunAsPPL /t REG_DWORD /d 1 /f
```

这条命令启用 RunAsPPL，让 LSASS 以受保护进程运行。重启后生效。

重启之后，我们再试一次 Mimikatz 的 `privilege::debug`。如果 RunAsPPL 生效了，Mimikatz 将无法读取 LSASS 内存——即使你有管理员权限。

这就是防御的核心思路：不是阻止攻击者获得管理员权限（那是密码策略的事），而是即使他有了管理员权限，也读不到凭据。纵深防御，层层设卡。

**关于 Credential Guard**，补充说明一下：它比 RunAsPPL 更强，用虚拟化技术把凭据完全隔离。但它需要 TPM 2.0 和 Secure Boot，我们的虚拟机环境通常不具备这些条件，所以这里只作为知识扩展了解，不实际配置。

---

## 总结与收尾（5 分钟）

好，我们来回顾一下今天走过的完整攻击链：

1. 从一个 IP 地址开始，通过 SMB 空会话拿到了用户名列表
2. 用户名列表加弱密码字典，暴力破解拿到了管理员凭据
3. 用管理员权限运行 Mimikatz，从 LSASS 内存中提取了所有人的哈希和明文密码
4. 用 Pass-the-Hash 证明了"有哈希就等于有密码"
5. 检测到了隐藏的后门账户
6. 最后配置了密码策略和 LSA 保护，验证了加固确实有效

每一步都建立在前一步的基础上，每一个攻击阶段都对应一个具体的安全弱点，每一个弱点都有对应的加固措施。这就是"攻防闭环"的思路。
